# 【设计】系统规则与设计方案（对标星露谷教训）（V1）

> 本文档为“可直接落地”的系统规则与设计方案汇总稿。每个模块均包含：
> 星露谷教训 → 我们的规则 → 具体方案 → 工程落点（目录/数据/存档预留）。
>
> 适用阶段：0.1.x（MVP 原型阶段 → 留存循环扩展阶段）

## 0. 总原则（不会轻易改）

- **P0 优先**：先打通闭环（茶→加工→经营→社交/战斗→回流），再扩展玩法面。
- **强反馈链**：任何“净化/加工/社交推进/节日结算”必须可被玩家在 1-2 天内感知。
- **数据驱动**：可见文案、节奏参数、表数据优先走 `configs/` + `assets/data/`，避免硬编码散落。
- **存档前向兼容**：新增系统先预留 tag 与 schema，读取侧可忽略未知字段。

## 一、强叙事分量：开场回忆 + 庄园日记系统

### 星露谷教训
爷爷的信只有开头一封，之后玩家与前任的情感联系断层；一次性的情感锚点撑不起长线共鸣。

### 我们的规则
**让前任庄园主成为“沉默的第四位 NPC”**：她/他已逝，但通过日记碎片、旧物、季节回忆持续在场。

### 具体方案

#### a) 开场回忆片段（30 秒，可跳过）
- 形式：像素动画/静态图序列（不强制重复播放）。
- 镜头脚本（建议最小实现版）：
  1) 一只手将紫砂壶放入木箱  
  2) 手写信浮现：“山庄交给你了。茶知道答案。”  
  3) 玩家推开庄园大门，画面黑白→彩色  

#### b) 庄园日记系统（触发解锁 + 可阅读）
- 入口：茶室书桌上的旧皮册（交互翻阅）。
- 核心：以**碎片式解锁**承载叙事与奖励（解锁=里程碑反馈）。

数据模型（设计稿）：

```cpp
struct DiaryEntry {
    int day_unlocked;
    std::string title;
    std::string text;
    std::string trigger_condition; // "FIRST_HARVEST" / "FULL_MOON" / "BEFRIEND_X" ...
    bool has_been_read;
};
```

触发条件（最小集）：
- 第一次收获：解锁“第一次种茶也手忙脚乱”
- 第一次净化浊灵：解锁“浊灵只是迷路的灵气”
- 满月之夜：解锁“月光位置”
- 与 NPC 好感达标：解锁“曾介绍我们认识”

完成奖励（示例）：
- 全部解锁后：庄园传承道具（如“旧茶筅”武器皮肤/外观）。

### 工程落点（建议）
- **domain**：`DiarySystem`（规则、触发判定、解锁状态）
- **infrastructure**：日记条目加载（JSON/CSV）；存档读写
- **engine**：交互入口（茶室书桌）、UI 面板渲染

### 存档预留（建议 tag）
- `diary_schema|v1|...`（可选，便于迁移）
- `diary|entry_id|day_unlocked|has_been_read`

## 二、云海垂钓系统

### 设计理念
不是“鱼换钱”，而是**垂钓即冥想**：钓起灵气的具象碎片，产出进入庄园循环（辅料/喂养/强化/恢复）。

### 具体方案

数据模型（设计稿）：

```cpp
enum class CloudSeaFish {
    CLOUD_CARP,
    MIST_JELLYFISH,
    DEW_SPIRIT,
    TIDE_SERPENT,
    LIGHT_SHARD,
};

struct FishingSpot {
    std::string map_id;
    int x, y;
    CloudSeaElement element_dominant;
    std::vector<FishSpawnEntry> fish_table;
    float bite_difficulty;
};
```

玩法要点（MVP 版）：
- 钓点：庄园小湖 / 后山云瀑 / 界桥边缘 / 瀑布下潭
- 季节：冬季云鲤更多，夏季霞水母频繁（表驱动）
- 上钩 QTE：极简节奏按键；失误不丢鱼但变慢（降低挫败）
- 产出回流：加工辅料 / 灵兽食物 / 强化材料 / 直接恢复体力

### 工程落点（建议）
- **domain**：`FishingSystem`（抛竿、咬钩、结算）
- **engine**：交互、QTE（可先做简化条形/节奏提示）
- **assets/data**：`assets/data/fishing/*.json`（钓点表、鱼表、季节权重）

### 存档预留（建议 tag）
- `fishing|skill_lv|...`（如果需要独立技能）
- `fishing_spot_state|map_id|x|y|cooldown|...`（可选）

## 三、地图 DIY 系统（茶室 + 庭院）

### 核心理念
把创造力出口从“农田布局”迁移到 **茶室 + 庭院 DIY**（玩家的创作画布）。

### 我们的规则
- 可编辑：庄园建筑内部（茶室布局）、庭院装饰、茶园小径方向
- 不可编辑：NPC 住所、界桥、野外（保证剧情空间一致性）
- 操作：布置模式（快捷键 Tab），物品从背包拖拽放置

### 具体方案

```cpp
struct PlacedObject {
    int object_id;
    int tile_x, tile_y;
    int rotation;       // 0/90/180/270
    std::string custom_data;
};
```

可放置物（首批）：
- 茶具架、屏风、盆栽、挂画、灵兽窝、灯笼、石径、室内盆栽茶树（小型可收获）
- 节日限定：春节红灯笼、秋节枫叶挂饰

### 工程落点（建议）
- **domain**：`DecorationPlacement`（格子占用、旋转规则）
- **engine**：布置模式输入、预览、放置回写
- **engine/rendering**：图层叠加（PlacedObject layer）
- **存档**：与已预留的 `decor_item` tag 对齐（见工程侧现有预留格式）

### 存档预留（建议 tag）
- `decor_item|id|x|y|rot|room|custom`（与工程侧预留保持一致）

## 四、茶叶加工链 + 食谱加工系统

### 设计规则
**金钱不是万能中介，灵气才是。** 加工链出口是“茶/食物”，同时具备：出售、赠送、自用 buff、战斗消耗品四用途。

### 茶叶加工链（规则）

```
鲜叶（采摘）
  ↓
萎凋（晾晒架，消耗晴天时间）
  ↓
杀青（炒茶锅，消耗体力）
  ↓
揉捻（揉茶台，小游戏或自动）
  ↓
干燥（烘焙笼，消耗灵气或柴火）
  ↓
成品灵茶（品质：净/雅/灵/圣）
```

品质波动来源：
- 工具等级、季节、技能、云海状态、灵兽协助、是否“欠损”惩罚等

### 食谱系统（规则）

```cpp
struct RecipeData {
    std::string id;
    std::string name;
    std::vector<Ingredient> ingredients;
    std::string result_item_id;
    int cook_time_minutes;
    std::vector<BuffEffect> buffs;
    std::string unlock_condition;
};
```

解锁来源：
- NPC 好感、节日、日记残页、探索宝箱

效果三类：
1) 恢复类：体力/行动力
2) 增益类：临时属性（战斗/探索）
3) 社交类：特定食物赠送特定 NPC 好感翻倍

### 工程落点（建议）
- **assets/data**：`assets/data/recipes/*.json`（配方表）
- **domain**：`RecipeSystem`（解锁、生产、消耗）
- **engine**：厨房/野餐点交互、UI
- **infrastructure**：数据加载 + 存档落盘

### 存档预留（建议 tag）
- `recipe_unlock|id`
- `processed_queue|machine_id|...`（若复用工坊队列可不新增）

## 五、社交系统深化：好感可见化 + 日程可视化 + 结婚助益

### 好感度可见化
- 默认不显示心级，避免“攻略感 UI”
- 在对话界面右上角用“云形态”表达阶段：
  - 薄云（0-3）相识
  - 层云（4-6）熟络
  - 霞云（7-9）知己
  - 七彩祥云（10）灵魂之友（专属边框/配色）

### 日程可视化（规则）
- 每个 NPC 按季节 4 套日程，天气为副条件
- 灵气潮日：全 NPC 聚集祈福（高价值社交日）
- 地图显示 NPC 大致位置需好感≥4（熟悉生活习惯）

### 结婚助益（以茶道替代生育）
- 结婚条件：满好感 + 专属剧情链 + “同心茶盏”（高茶艺加工品）
- 婚后助益二选一（可切换）：
  - 伴侣采茶：每日自动收获 30%
  - 伴侣制茶：每日自动加工一次（品质略低但省时）
- 每周双人品茶事件：维持好感 + 触发回忆对话

### 工程落点（建议）
- 关系状态机与婚后 buff：复用现有 `RelationshipSystem`（已实现基础）
- 日程与可视化：复用现有 NPC 日程系统文档与数据表（需在 UI 层做呈现）

## 六、技能系统：分支天赋

### 规则
- 10 级出现分支（A/B），以“玩法风格”而非纯数值堆叠区分
- 分支应与四大循环绑定（经营/社交/探索/战斗）

（表略：与原提案一致，后续落入技能表数据与 UI）

### 工程落点（建议）
- **assets/data**：技能树 JSON（分支、解锁条件、效果）
- **domain**：技能效果解算
- **engine**：技能 UI/选择

---

## 七、季节节庆系统（四季节日）

### 规则
- 每季至少一个“记忆锚点节日”
- 节日奖励必须能推动至少一条长期线（技能/配方/装饰/关系）

（春茶祭/灵兽祭/丰穰宴/界桥祭：按原提案落地为节日卡 + 事件脚本）

### 工程落点（建议）
- 复用现有 Festival 系统：节日数据表 + 触发事件 + 自动奖励（已具备框架）

## 八、架构预留：MOD 友好数据入口

### 规则
- 所有核心数据通过统一注册表加载（JSON），启动时校验并输出 hash
- MOD 通过替换/追加 JSON 实现改动，冲突可追溯

建议目录约定：
- `assets/data/crops/*.json`
- `assets/data/weapons/*.json`
- `assets/data/npcs/*.json`
- `assets/data/recipes/*.json`
- `assets/data/festivals/*.json`

## 九、核心优势可视化：净化回流庄园（必须可见）

### 规则
“净化→庄园收益”不能是隐藏数值，必须形成 UI/视觉/次日结算三段反馈。

### 具体方案（最小实现）
1) 战斗结算通知：灵气结晶 +N，茶园/云海受益提示  
2) 次日清晨：提示“昨日净化，今日生机盎然”  
3) 视觉：1-3 天茶园灵气光点更明亮  
4) 品质联动：茶园灵气影响茶叶基础品质，让玩家感到价值

## 十、整体优先级（执行建议）

- **P0**：茶叶加工链；战斗净化回流可视化
- **P1**：社交好感可见 + 日程；四季节日（记忆锚点）
- **P2**：技能分支；庄园日记系统
- **P3**：云海垂钓；地图 DIY（先茶室）
- **P4**：MOD 友好 JSON 架构（先预留接口与校验机制）
