# 【系统】动态叙事与行为记忆（V1.0）

**日期**：2026-04-28  
**状态**：设计草稿 / 可进入工程拆分  
**定位**：让 NPC、邮件与局部事件持续“记住”玩家最近的生活方式，强化陪伴感与被世界看见的治愈体验

---

## 1. 设计目标

### 1.1 核心问题

当前社交系统已有好感、日程、事件与关系推进骨架，但 NPC 对玩家近期行为的感知较弱，容易形成：

- 对话只体现“关系阶段”，不体现“最近发生了什么”
- 玩家完成高价值行为后，情绪反馈主要来自数值与 UI，缺少角色层面的回应
- 邮件与日常问候容易变成固定脚本，长线游玩时陪伴感递减

### 1.2 系统目标

- 让 NPC 能对玩家最近 3-7 天的重要行为作出自然回应
- 让邮件系统从固定奖励投递升级为“情感回声”载体
- 让社交反馈贴合《云海山庄》的日常节奏，不制造任务焦虑
- 为后续恋爱、节日、庄园生态、茶灵收集提供统一叙事钩子

### 1.3 设计原则

- **轻追踪**：只记录少量高价值行为，不追踪一切
- **弱打扰**：反馈以闲聊、信件、附带评论为主，不弹强提醒
- **重语气**：同一事件由不同 NPC 给出不同情绪角度，而非统一播报
- **可衰减**：行为记忆会自然淡出，避免对话池被陈旧内容污染

---

## 2. 核心体验循环

```text
玩家生活行为
  -> 行为被写入“行为记忆”
  -> 系统筛选近期可回应话题
  -> NPC 对话 / 邮件 / 小事件引用该话题
  -> 玩家感到“世界记得我做过的事”
  -> 更愿意持续经营、社交与尝试新玩法
```

### 2.1 典型触发示例

1. 玩家连续 3 天都在后山采集或战斗
2. 行为记忆系统写入“近期频繁探索后山”
3. 次日与 `晚星` 对话时触发关心型评论，与 `老槐` 对话时触发经验型评论
4. 若玩家与特定角色好感较高，晚间还可能收到一封提醒休息的信

---

## 3. 行为记忆模型

### 3.1 追踪范围

首版只追踪对情绪反馈价值高、又能稳定判定的事件：

| 类别 | 示例事件 | 用途 |
|---|---|---|
| 茶事成就 | 制出圣品灵茶、首次完成某高品质茶 | 激发赞叹、请茶、师徒点评 |
| 经营节奏 | 连续多日忙于农田/工坊/茶室 | 体现 NPC 对玩家生活状态的观察 |
| 探索行为 | 连续前往后山、净化浊灵、参加界桥活动 | 触发关心、敬佩或剧情延展 |
| 社交行为 | 高频送礼、参加节日、触发重要关系节点 | 强化关系反馈闭环 |
| 生态行为 | 庄园元素明显失衡或趋于平衡 | 让 NPC 对庄园气息变化发表评论 |
| 收集行为 | 获得新茶灵、图鉴里程碑 | 触发稀有评论与收藏讨论 |

### 3.2 数据结构建议

```cpp
enum class PlayerMemoryType {
    BrewHolyTea,
    BrewNewTea,
    FarmFocused,
    WorkshopFocused,
    ExploreBackMountain,
    PurifyMiasma,
    FestivalAttend,
    RelationshipMilestone,
    EcoBalanced,
    EcoImbalanced,
    TeaSpiritUnlocked
};

struct PlayerMemoryRecord {
    PlayerMemoryType type;
    std::string subject_id;       // tea_id / npc_id / map_id / element_id
    int game_day = 0;
    int weight = 1;               // 强度，用于排序
    int decay_days = 7;           // 记忆衰减时长
    bool consumed_by_mail = false;
    bool consumed_by_dialogue = false;
};
```

### 3.3 聚合规则

- 相同类型、相同 `subject_id` 的重复行为可叠加 `weight`
- 同一日发生的高频行为不逐条存储，统一合并为一条摘要
- 超过 `decay_days` 的记录自动失效，不再参与抽取
- 单日最多向对话系统提供 2 个热点话题，避免信息噪声

## 4. 对话生成与投放规则

### 4.1 对话钩子结构

```cpp
struct DialogueMemoryHook {
    std::string hook_id;
    PlayerMemoryType memory_type;
    std::string speaker_id;
    int min_heart_level = 0;
    int max_days_since_memory = 7;
    int priority = 0;
    std::string season_limit;
    std::string weather_limit;
    std::string text_template;
};
```

### 4.2 抽取优先级

1. 最近 2 天内发生的高权重事件
2. 与当前 NPC 身份高度相关的事件
3. 与当前天气、季节、地点相契合的事件
4. 未被当前 NPC 用过的事件

### 4.3 文案风格约束

- 不直接读系统日志，不说“系统检测到你……”
- 更像听闻、观察与陪伴，例如：
  - “听说你最近把茶火候守得很稳，整个茶室都是香的。”
  - “这几日总在后山看见你，云雾虽好，也记得歇脚。”
  - “庄园里的风声顺了不少，你最近是不是把灵气调匀了？”

### 4.4 角色差异化

同一记忆类型应按 NPC 气质拆开：

- `阿茶`：偏日常、温暖、会主动请喝茶
- `晚星`：偏安静观察、含蓄关心
- `老槐`：偏经验与长者点评
- `玄清`：偏修行视角，强调气息与心境

## 5. 邮件系统联动

### 5.1 邮件定位调整

邮件不再只承担奖励投递，还承担：

- 关系升温后的主动关心
- 对玩家近期行为的延迟回应
- 为重要系统提供“低打扰二次反馈”

### 5.2 邮件触发建议

| 触发条件 | 邮件类型 | 说明 |
|---|---|---|
| 玩家制出首个圣品茶 | 请茶/称赞信 | 邀请带茶来坐坐，推动二次互动 |
| 连续 3 天高强度探索 | 关心信 | 提醒休息，附带小礼物或补给 |
| 生态首次稳定平衡 | 观察信 | NPC 描述庄园气息变化 |
| 新茶灵加入图鉴 | 传闻信 | 某 NPC 听闻后表达好奇或提供线索 |

### 5.3 投递规则

- 同一类“关心邮件”一周最多 1 封
- 关系较浅时只触发闲聊，不触发私信
- 邮件内容以抒情与陪伴为主，奖励为辅

## 6. 与现有系统的耦合点

### 6.1 domain 层

- 新增 `PlayerMemorySystem`
- 接收来自以下系统的事件输入：
  - `RelationshipSystem`
  - `WorkshopSystem`
  - 茶叶/灵茶品质结算
  - 节日结算
  - 庄园生态系统
  - 茶灵图鉴系统

### 6.2 engine 层

- 在每日结算后进行记忆衰减与热点刷新
- 对话运行时进入 NPC 对话前，向 `PlayerMemorySystem` 请求候选钩子
- 邮件生成器在次日清晨或固定时段拉取可投递事件

### 6.3 infrastructure 层

- 增加记忆钩子表加载
- 增加邮件模板表加载
- 存档增加行为记忆持久化

### 6.4 app 层

- 无需新增大状态，仅在日切/邮件入口增加调度调用

## 10. 风险与约束

### 10.1 主要风险

- 若追踪事件太多，会让文案池膨胀、维护成本过高
- 若触发过于频繁，会削弱稀缺感并造成重复
- 若文案过于直白，会破坏世界自然感

### 10.2 约束策略

- 首版仅保留 6-10 个高价值记忆类型
- 单 NPC 每日最多消费 1 个动态记忆钩子
- 同类文案必须至少准备 3 条变体再上线

---

## 11. 验收标准

- [ ] 玩家完成“制出圣品茶”后，48 小时内至少触发 1 条相关评论或邮件
- [ ] 连续 3 天探索后山后，可稳定触发关心型反馈
- [ ] 不同 NPC 对同一事件存在明显语气差异
- [ ] 读档后未消费的记忆记录仍可继续生效
- [ ] 无行为记忆时，对话系统可安全回退到普通台词
