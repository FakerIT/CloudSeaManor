# 《云海山庄》完整对标开发任务拆解卡

## 文档目标
基于“完整对标功能清单”，把《云海山庄》的长期开发拆成可执行任务层，作为现有 8 周 MVP 任务卡之后的中长期开发参考。

本卡重点回答三件事：
1. 哪些功能要做
2. 按什么顺序做
3. 每个阶段的交付物与验收标准是什么

---

## 总开发阶段划分

### Phase 0：现有原型收束期
目标：把当前 Demo 原型整理成可稳定扩展的代码底座。

### Phase 1：P0 核心闭环完善期
目标：完成真正可连续游玩的山庄经营闭环。

### Phase 2：P1 系统扩展期
目标：让农业、社交、节日、探索、灵兽形成多路径成长。

### Phase 3：P2 长期内容期
目标：建设长期目标、完美度、家庭、后期挑战与内容扩展能力。

---

## Phase 0：现有原型收束期（建议 3-5 周）

### P0-1 代码结构整理
**目标**：继续拆分当前 `GameApp.cpp` 巨文件，形成农业 / 日切 / HUD / 灵兽 / NPC / 工坊模块边界。

**任务**：
- 拆分作物系统规则层、表现层、数据层
- 拆分灵兽行为状态与协助逻辑
- 拆分 NPC 日程 / 对话 / 送礼模块
- 拆分工坊加工逻辑与 UI 面板数据
- 收口存档恢复入口

**产出物**：
- 独立模块文件结构
- 降低 `GameApp.cpp` 体积
- 各系统具备明确输入输出边界

**验收**：
- 主循环仍稳定可运行
- 不出现功能回归
- 存读档与日切逻辑无破坏

---

### P0-2 数据驱动底座
**目标**：把高频玩法参数从硬编码迁移到配置或数据表。

**任务**：
- 作物成长参数表
- 云海倍率表
- NPC 日程表
- 礼物偏好表
- 配方表
- 建筑升级参数表
- 节日基础表

**产出物**：
- `configs/` + `assets/data/` 数据表结构规范
- 统一数据加载与校验流程

**验收**：
- 至少 5 类核心玩法参数不再硬编码
- 参数改动可直接影响运行结果

---

### P0-3 UI / 手账信息底座
**目标**：把现有 HUD 提升为可扩展的信息系统。

**任务**：
- 拆出 HUD 文本构建器
- 增加手账入口
- 增加日期、生日、节日、预报、今日目标面板
- 明确“推荐任务 / 当前目标 / 明日预告”三层信息

**产出物**：
- HUD 模块
- 手账原型
- 日历原型

**验收**：
- 新玩家 10 分钟内能理解核心循环
- 关键状态无需看调试标题也能明白

---

## Phase 1：P0 核心闭环完善期（建议 8-12 周）

### P1-1 日循环与云海中枢正式版
**目标**：形成完整每日循环节奏。

**任务**：
- 22:00 可休息、02:00 强制结算
- 每日重置规则统一
- 云海预报与特殊大潮提示
- 每日事件刷新入口
- 疲劳 / 低体力反馈强化

**功能点**：
- 时间推进
- 睡觉结算
- 次日预报
- 每日目标推荐

**验收**：
- 一天可稳定流逝到结算
- 日切不会导致系统状态错乱

---

### P1-2 农业与灵茶系统正式版
**目标**：把现有最小耕种原型升级为可扩展农业系统。

**任务**：
- 多作物数据驱动
- 品质系统（普通 / 优质 / 稀有 / 灵品）
- 云海对成长、品质、产量的正向影响
- 肥料与土壤加成
- 稀有作物获取入口

**功能点**：
- 多阶段成长
- 多作物差异参数
- 品质掉落
- 稀有种子

**验收**：
- 至少 5 种可种植内容
- 浓云 / 大潮环境下成长与品质差异明显

---

### P1-3 工坊与经济闭环正式版
**目标**：把农业产出稳定转化为加工收益与山庄升级收益。

**任务**：
- 工坊队列加工
- 多类配方（茶类 / 料理 / 建材 / 灵器）
- 售卖 / 订单 / 收益反馈
- 山庄升级所需材料链

**功能点**：
- 原料转工艺品
- 收益对比反馈
- 建筑解锁新经济链

**验收**：
- 农业 → 工坊 → 售卖 / 升级闭环稳定
- 玩家能感知加工比卖原料更优

---

### P1-4 基础社交系统正式版
**目标**：建立“对话 + 送礼 + 好感推进 + 日程变化”完整社交骨架。

**任务**：
- 时间段差异对话
- 地点差异对话
- 礼物偏好多维反馈
- 基础心级事件触发
- 日程与天气 / 节日联动

**功能点**：
- 可送礼
- 可升好感
- 可触发事件

**验收**：
- 至少 3 位角色具备完整送礼 / 对话 / 心级推进体验

---

### P1-5 灵兽协助系统正式版
**目标**：让灵兽真正成为经营协助者，而不是装饰性存在。

**任务**：
- 灵兽互动 / 心情 / 协助次数
- 农业辅助、采集辅助、恢复辅助
- 基础成长阶段
- 灵兽居住点与日常活动范围

**功能点**：
- 每日互动
- 协助行为
- 状态成长

**验收**：
- 灵兽至少能对两个核心系统提供稳定帮助

---

### P1-6 存档、手账、日历正式版
**目标**：保证连续游玩与计划感。

**任务**：
- 多存档位
- 手账记录成熟 / 节日 / 生日 / 预报
- 周目标 / 日目标 / 已完成事项
- 存档版本号与迁移预留

**功能点**：
- 连续经营
- 节日计划
- 成熟提醒

**验收**：
- 玩家连续游玩 7 天无阻断
- 信息不依赖记忆也能管理

---

## Phase 2：P1 系统扩展期（建议 12-20 周）

### P2-1 多地图与建筑互连收益系统
**目标**：增强家园成长感与长期规划空间。

**任务**：
- 山庄区域功能分化
- 主屋扩建
- 茶园 / 工坊 / 客栈 / 灵兽园 / 温泉区联动收益
- 建筑升级影响多个系统

**验收**：
- 每次建筑升级至少影响 2 个以上系统

---

### P2-2 节日系统正式版
**目标**：建立全年节奏感与社交高峰节点。

**任务**：
- 12 个核心节日骨架
- 节日前 3 天预告
- 节日小游戏 / 奖励 / 限定对话
- 节日与经济、送礼、山庄评价联动

**验收**：
- 至少 4 个节日有完整可玩内容

---

### P2-3 钓鱼 / 采集 / 探索三分支补完
**目标**：让玩家除农业外还有稳定成长路径。

**任务**：
- 云海垂钓
- 季节 / 时间 / 天气采集
- 灵界遗迹资源回收
- 图鉴与收集进度

**验收**：
- 非农业玩家也能获得稳定收益与成长感

---

### P2-4 社交深化与恋爱系统
**目标**：把“能送礼”扩展为“能长期关系推进”。

**任务**：
- 多心级事件
- 雨天 / 节日 / 生日 / 深夜 / 特殊天气剧情
- 恋爱确认与关系转折
- 婚后协助雏形

**验收**：
- 至少 2 位角色具备完整恋爱推进链

---

### P2-5 技能与长期成长系统
**目标**：建立长期留存动力。

**任务**：
- 农业 / 采集 / 探索 / 社交成长等级
- 阶段解锁配方与能力
- 类 Mastery 永久成长原型
- 云海守护契约入口

**验收**：
- 满级后仍有长期成长空间

---

## Phase 3：P2 长期内容期（建议 20 周+）

### P3-1 家庭与婚后生活系统
**任务**：
- 结婚后共同生活
- 配偶帮忙
- 家庭事件
- 孩子或家庭成长线

**优先级**：中

---

### P3-2 灵兽多只编组与进化繁殖
**任务**：
- 多灵兽养成
- 进化路线
- 协助分工
- 特殊组合收益

**优先级**：中

---

### P3-3 山庄评价、完美度、收集馆
**任务**：
- 图鉴 / 文物馆 / 节日完成度 / 山庄评价
- 完美经营目标
- 长期奖励与荣誉系统

**优先级**：中

---

### P3-4 深度探索与可选战斗模式
**任务**：
- 灵界深层挑战
- 关闭战斗 / 轻挑战 / 深挑战三模式
- 稀有掉落与长线收集

**优先级**：中

---

**任务**：
- JSON 内容包

- 官方内容模板与示例

**优先级**：高（架构层面必须早预留，正式开放可后置）

## 功能点总任务映射

### 高优先级功能簇
1. 日循环 / 睡觉 / 预报 / 手账
2. 农业 / 灵茶 / 品质 / 云海加成
3. 工坊加工 / 售卖 / 经济
4. NPC 对话 / 送礼 / 心级
5. 灵兽协助
6. 存档与 UI 反馈

### 中优先级功能簇
1. 节日
2. 建筑互连收益
3. 钓鱼 / 探索 / 采集补充分支
4. 长期成长与契约卷册
5. 家庭与婚后系统

### 低优先级 / Stretch Goal
1. 联机
2. 完整 Mod 开放生态
3. 大规模后期挑战塔 / 无尽模式

## 推荐排期策略

### 先做
- 当前原型结构整理
- 多作物与工坊正式版
- 基础社交正式版
- 手账与日历

### 再做
- 节日
- 采集 / 钓鱼 / 探索分支
- 建筑互连与中期成长

### 最后做
- 家庭
- 完美度
- 深层挑战
- 联机与开放 Mod 工具

---

## 每阶段统一验收问题
每增加一个系统，都必须回答：

1. **它和哪个现有系统形成闭环？**
2. **玩家是否能在 1-2 天内感知到收益？**
3. **它是否已经接入存档、日切、UI 提示？**
4. **它是否可以配置化 / 数据驱动？**
5. **它是否强化了“云海山庄”的差异化，而不是只是复制 SDV？**

如果 5 个问题中有 2 个以上答不上来，则该功能不应立即进入开发。

---

## 结论
这份任务拆解卡的核心目标不是“把 SDV 全做一遍”，而是：
- 先保留 SDV 被验证过的强循环骨架
- 再把《云海山庄》的云海、灵茶、灵兽、守护任务做成自己的长期优势
- 用阶段化任务避免系统越堆越散、代码越写越乱
