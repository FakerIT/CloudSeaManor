# MVP执行层Issue拆解卡（Week1-Week8）

## 使用规则
- 优先级：`P0` 必做，`P1` 重要，`P2` 优化
- 前置依赖：表示开始该 Issue 前必须完成的内容
- 验收点：完成标准，必须可验证

---

## Week1：项目地基与可运行原型

### 优化空间
- 第一次进入场景时，玩家容易不知道“我现在能做什么”
- 移动、碰撞、交互如果反馈偏弱，会直接影响第一印象
- HUD 如果只为开发服务，玩家会看到信息却不知道信息的用途

### 如何提升用户体验
- 把首屏 10 秒体验做顺：进入场景后立即知道怎么移动、哪里能互动、当前目标是什么
- 让移动、碰撞、转向反馈稳定一致，优先保证“手感可信”
- HUD 先只保留时间、体力、提示这类高价值信息，避免认知负担
- 给第一个交互点更强的视觉引导，帮助玩家快速建立操作信心

### Issue 列表
1. 初始化工程与运行窗口
- 优先级：P0
- 前置依赖：无
- 验收点：启动后稳定进入主窗口，无黑屏、无立即崩溃

2. 建立主循环与事件处理骨架
- 优先级：P0
- 前置依赖：初始化工程与运行窗口
- 验收点：`Run / ProcessEvents / Update / Render` 正常运转，关闭事件可退出

3. 完成玩家移动与基础碰撞
- 优先级：P0
- 前置依赖：主循环骨架
- 验收点：WASD 可移动，玩家不会穿过主要障碍

4. 接入场景占位与交互点原型
- 优先级：P1
- 前置依赖：玩家移动与碰撞
- 验收点：场景内至少存在房屋、障碍物、1 个交互点

5. 搭建最小 HUD 与提示栏
- 优先级：P1
- 前置依赖：主循环骨架
- 验收点：界面可显示时间、体力、提示文字等信息

6. 接入日志输出
- 优先级：P1
- 前置依赖：初始化工程与运行窗口
- 验收点：启动、退出、基础交互有日志记录

7. 补首屏引导反馈
- 优先级：P2
- 前置依赖：HUD 与提示栏
- 验收点：首次进入场景时玩家可看到移动/交互提示

## Week2：种田最小闭环

### 优化空间
- 玩家最容易卡住的不是“不会按键”，而是不知道地块现在处于什么状态
- 如果播种、浇水、成长、收获之间缺少明显反馈，闭环会显得像重复劳动
- 背包变化如果不可感知，玩家会感到产出没有价值

### 如何提升用户体验
- 把地块状态可视化做清楚，让玩家一眼知道“下一步该做什么”
- 每次关键操作都给立刻反馈，形成短正反馈循环
- 失败提示要说清原因，例如缺种子、未浇水、尚未成熟，而不是笼统失败
- 让收获入包和数量变化足够明显，强化“劳动有结果”的感受

### Issue 列表
1. 创建固定地块与地块状态机
- 优先级：P0
- 前置依赖：Week1 场景原型
- 验收点：每块地拥有未开垦、已播种、已浇水、已成熟等状态

2. 接入翻地与播种交互
- 优先级：P0
- 前置依赖：地块状态机
- 验收点：玩家可翻地并在拥有种子时播种

3. 接入浇水与收获交互
- 优先级：P0
- 前置依赖：翻地与播种交互
- 验收点：玩家可浇水并对成熟作物收获

4. 建立作物成长值与阶段显示
- 优先级：P1
- 前置依赖：浇水与收获交互
- 验收点：作物随时间推进成长，至少 3 段可感知变化

5. 打通背包消耗与产出闭环
- 优先级：P1
- 前置依赖：翻地与播种交互
- 验收点：播种消耗种子，收获增加作物数量

6. 补种田失败与成功反馈
- 优先级：P1
- 前置依赖：浇水与收获交互
- 验收点：未满足条件时有明确提示，成功时有即时反馈

7. 完成 3 次连续种植测试
- 优先级：P2
- 前置依赖：成长值与阶段显示
- 验收点：至少完成 3 次完整种植闭环，无状态错乱

## Week3：天气与生长联动

### 优化空间
- 只有天气名称变化而没有结果差异时，玩家会把天气系统视为装饰
- 如果天气预报没有决策价值，它只会占 HUD 空间
- 环境变化太弱会让“云海变化”这个卖点失去记忆点

### 如何提升用户体验
- 让天气既能被看到，也能被用到：看到是氛围，用到是策略
- 预报信息要能支持玩家做简单判断，例如今天该重点种植还是等待
- 天气切换反馈要短、快、明确，不打断操作但能被感知
- 不同天气影响应有区别，但不能让玩家觉得被随机惩罚

### Issue 列表
1. 定义天气枚举与切换规则
- 优先级：P0
- 前置依赖：Week2 种田闭环完成
- 验收点：至少 3 种天气可稳定切换并被读取

2. 接入天气对作物成长倍率影响
- 优先级：P0
- 前置依赖：天气切换规则
- 验收点：不同天气下成长速度有明确差异

3. 补充天气 HUD 与预报展示
- 优先级：P1
- 前置依赖：天气切换规则
- 验收点：HUD 可显示当前天气与至少 1 日预报

4. 实现环境氛围反馈
- 优先级：P1
- 前置依赖：天气切换规则
- 验收点：背景色或氛围效果能体现天气差异

5. 建立天气切换提示文案
- 优先级：P1
- 前置依赖：天气 HUD 与预报
- 验收点：天气变化时玩家能立即收到清晰提示

6. 完成跨天气种植测试
- 优先级：P2
- 前置依赖：天气成长倍率
- 验收点：同一作物在不同天气下完成至少 1 轮对比验证

## Week4：工坊与基础经营

### 优化空间
- 如果加工只是“按一下然后等结果”，经营层会显得很薄
- 玩家若不知道加工产物有什么用，就不会形成资源规划意识
- 资源短缺若没有明确提示，会被误解成系统坏了或操作无效

### 如何提升用户体验
- 让工坊具备过程感：启动、运行、完成都应该被玩家感知
- 让资源消耗和收益去向清楚可追踪，帮助玩家理解经营意义
- 让建设或修缮目标挂在玩家眼前，形成中期目标牵引
- 在失败时说明“差什么”，在成功时说明“得到了什么”

### Issue 列表
1. 接入工坊对象与交互入口
- 优先级：P0
- 前置依赖：Week3 天气系统稳定
- 验收点：玩家可接近工坊并触发加工交互

2. 建立原料消耗与成品产出规则
- 优先级：P0
- 前置依赖：工坊对象与交互入口
- 验收点：加工正确消耗原料并生成成品

3. 补加工进度与完成反馈
- 优先级：P1
- 前置依赖：原料消耗与成品产出
- 验收点：运行中可看到进度，完成时有提示

4. 接入主屋修缮或资源消耗目标
- 优先级：P1
- 前置依赖：原料消耗与成品产出
- 验收点：至少存在 1 个可被资源推进的建设目标

5. 补资源不足与失败反馈
- 优先级：P1
- 前置依赖：工坊对象与交互入口
- 验收点：资源不足时会提示具体缺失资源

6. 验证经营节奏闭环
- 优先级：P2
- 前置依赖：资源消耗目标
- 验收点：完成至少 2 次“种植 -> 加工 -> 消耗”闭环测试

## Week5：灵兽互动雏形

### 优化空间
- 灵兽如果只有互动入口，没有实际帮助，玩家不会建立伙伴感
- 帮助效果太弱或不可见时，灵兽会退化成装饰物
- 每日互动若只是完成打卡，情感价值会很低

### 如何提升用户体验
- 让灵兽互动兼具情绪反馈和实用反馈，既可爱又有用
- 让帮助行为产生可见结果，例如直接影响种植效率或日常便利
- 让玩家清楚知道“今天灵兽有没有帮过我”，减少状态不确定感
- 灵兽移动与存在方式要轻量自然，避免抢镜但要有生命感

### Issue 列表
1. 创建灵兽实体与基础状态机
- 优先级：P0
- 前置依赖：Week4 经营闭环稳定
- 验收点：灵兽可在待机/游走/跟随间切换

2. 接入玩家与灵兽互动入口
- 优先级：P0
- 前置依赖：灵兽实体与状态机
- 验收点：玩家接近后可互动，互动后有明确反馈

3. 实现每日互动状态记录
- 优先级：P1
- 前置依赖：互动入口
- 验收点：每日互动状态可记录并在新一天重置

4. 实现 1 个灵兽辅助能力
- 优先级：P1
- 前置依赖：每日互动状态记录
- 验收点：辅助能力会真实影响至少 1 条日常循环

5. 补灵兽视觉与 HUD 反馈
- 优先级：P1
- 前置依赖：互动入口
- 验收点：灵兽状态、互动结果、今日帮助可被玩家感知

6. 完成每日重置联动测试
- 优先级：P2
- 前置依赖：灵兽辅助能力
- 验收点：连续跨 3 天测试中灵兽状态无残留问题

## Week6：NPC 与简单社交反馈

### 优化空间
- 只有对话按钮和送礼按钮，不足以让玩家感受到“关系”
- NPC 缺乏差异和位置变化时，世界会显得静态且假
- 好感变化如果没有清晰反馈，玩家无法理解送礼的意义

### 如何提升用户体验
- 让不同 NPC 在性格、反馈、喜好上立刻区分开
- 让送礼结果直接可见，减少“数值变化但没感觉”的问题
- 让 NPC 至少有轻量位置变化或日程感，增强生活感
- 让经营产出流向社交行为这件事变得合理且值得

### Issue 列表
1. 创建至少 2 个 NPC 实体与基础位置逻辑
- 优先级：P0
- 前置依赖：Week5 灵兽互动完成
- 验收点：场景中至少 2 个 NPC 可被找到并接近互动

2. 接入基础对话显示
- 优先级：P0
- 前置依赖：NPC 实体与位置逻辑
- 验收点：玩家交互后可看到基础对话反馈

3. 接入送礼与偏好反馈
- 优先级：P1
- 前置依赖：基础对话显示
- 验收点：不同 NPC 对同一礼物可给出不同反馈

4. 建立好感数值与每日送礼限制
- 优先级：P1
- 前置依赖：送礼与偏好反馈
- 验收点：送礼可增减好感，且每日限制正确工作

5. 补社交结果提示与 UI 反馈
- 优先级：P1
- 前置依赖：好感数值与每日送礼限制
- 验收点：送礼结果、好感变化、当日限制都有明确提示

6. 完成跨日社交稳定性测试
- 优先级：P2
- 前置依赖：好感数值与每日送礼限制
- 验收点：连续 3 天送礼 / 对话后无异常残留

## Week7：连续游玩测试与数值修正

### 优化空间
- 只做感性试玩而不结构化记录，很多问题会重复出现
- 如果不先区分阻断问题和体验问题，优化效率会很低
- 数值调整没有依据时，体验会在“更好一点”和“更糟一点”之间来回摇摆

### 如何提升用户体验
- 先修会让玩家中断、卡住、误解系统的关键问题
- 用测试辅助把“不好玩”的原因可视化，而不是靠印象判断
- 把体力、成长、产出、社交反馈节奏拉到顺滑区间，减少挫败感
- 让连续游玩的体验更稳定，确保 Demo 像一个整体而不是功能堆叠

### Issue 列表
1. 完成标准 7 日连续游玩记录
- 优先级：P0
- 前置依赖：Week6 社交闭环稳定
- 验收点：形成至少 1 份按天记录的问题列表

2. 修复高优先级阻断问题
- 优先级：P0
- 前置依赖：7 日连续游玩记录
- 验收点：影响主循环推进的阻断问题被修复或稳定记录

3. 补关键状态测试辅助信息
- 优先级：P1
- 前置依赖：7 日连续游玩记录
- 验收点：时间、天气、灵兽、NPC、存档等关键状态可被快速观察

4. 完成第一轮核心数值调整
- 优先级：P1
- 前置依赖：高优先级阻断问题修复
- 验收点：体力、成长、加工、送礼等关键数值进入可试玩区间

5. 整理高 / 中 / 低优先级问题清单
- 优先级：P1
- 前置依赖：7 日连续游玩记录
- 验收点：形成 1 份清晰的 Issue List / Backlog

6. 完成一轮修复后回归验证
- 优先级：P2
- 前置依赖：核心数值调整
- 验收点：修复后再次连跑 3 日以上，无明显回归性阻断问题

## Week8：存档、稳定性与 Demo 收口

### 优化空间
- 如果只修 bug 不整理演示体验，最终 Demo 仍会显得松散
- 存档系统即使可用，只要反馈不可信，玩家依然不敢依赖
- 没有清晰的版本边界和已知问题说明，演示时会频繁被动解释

### 如何提升用户体验
- 让存档、读档、自动保存都具备明确且可信的反馈闭环
- 让 Demo 演示顺序自然流畅，减少切换和解释成本
- 让已知问题、当前边界、后续方向表达清楚，降低预期落差
- 把玩家注意力留在玩法感受上，而不是异常和不确定性上

### Issue 列表
1. 完成关键状态存档完整性校验
- 优先级：P0
- 前置依赖：Week7 高优先级问题清单
- 验收点：时间、位置、背包、茶田、工坊、灵兽、NPC 等关键状态可稳定恢复

2. 补存档失败与异常存档反馈
- 优先级：P0
- 前置依赖：关键状态存档完整性校验
- 验收点：无存档、坏存档、读取失败时有清晰提示

3. 修复 Demo 阶段剩余高优先级体验问题
- 优先级：P1
- 前置依赖：Week7 高优先级问题清单
- 验收点：影响演示流畅度的关键问题被修复或明确记录

4. 完成 Demo 标准流程回归测试
- 优先级：P1
- 前置依赖：剩余高优先级体验问题修复
- 验收点：按标准链路可完整演示，无严重阻断

5. 整理 Demo 演示说明与版本边界
- 优先级：P1
- 前置依赖：Demo 标准流程回归测试
- 验收点：形成 1 份简洁清楚的演示说明与版本说明

6. 整理 Issue List 与后续 Backlog
- 优先级：P2
- 前置依赖：剩余高优先级体验问题修复
- 验收点：形成后续迭代可执行清单，并区分必须修与可延后项

## 下一阶段计划：Phase 2 结构升级方向

> 以下内容不再属于 MVP 收口优化，而属于下一阶段的系统级扩展与结构重构，应按独立阶段规划、独立验收、独立数据结构推进。

### 1. 更完整的作物系统
#### 目标
从当前“单一闭环作物原型”升级为“可扩展农业系统”，让不同作物具备明确差异、品质结果和成长策略。

#### 核心范围
- 建立多作物配置表与差异化数值
- 接入真正的品质变化逻辑
- 接入更复杂的成长条件与阶段规则
- 让天气、浇水、灵气、品质之间形成联动

#### 关键拆解
1. 多作物差异数值
- 生长时间
- 产量
- 售价
- 体力消耗
- 季节 / 天气适配

2. 真正的品质变化
- 普通 / 优质 / 稀有等品质层级
- 品质受天气、照料、收获时机影响
- 品质影响售价、加工收益、送礼效果

3. 更复杂成长逻辑
- 阶段式成长条件
- 缺水 / 延迟收获 / 连续照料影响
- 特殊作物或灵茶的额外成长规则

#### 验收方向
- 至少 4 种作物具备明确差异
- 品质系统会真实改变收益或用途
- 成长逻辑不再只是单一线性增长

### 2. 更完整的天气表现
#### 目标
将天气从“数值倍率 + tint”升级为“可见、可感知、可影响决策”的环境系统。

#### 核心范围
- 增加粒子层表现
- 增加云海动态层
- 增强场景随天气变化的视觉差异
- 让天气反馈不只在 HUD 中出现，而是进入世界本身

#### 关键拆解
1. 粒子层
- 雾气 / 漂浮灵尘 / 雨丝 / 光斑

### 3. 更完整的工坊体验
#### 目标
将当前“单配方、单次加工”升级为“多产线选择 + 经营规划”的轻经营系统。

#### 核心范围
- 增加多配方
- 增加队列加工
- 增加更完整的工坊 UI 面板
- 让加工成为资源规划的一部分，而不是一次性交互

#### 关键拆解
1. 多配方
- 茶类加工
- 料理类加工
- 基础建材 / 工艺品加工
- 配方解锁与用途差异

2. 队列加工
- 排队生产
- 加工中 / 待加工 / 已完成状态区分
- 多批次产物结算

3. 更完整 UI 面板
- 配方列表
- 原料需求显示
- 队列状态显示
- 完成产物与收益反馈

#### 验收方向
- 至少 5 个有效配方可运行
- 队列加工可稳定处理多批任务
- 工坊 UI 能支持玩家清楚规划输入和输出

### 4. 更完整的灵兽系统
#### 目标
将当前“单次互动 + 单技能辅助”升级为“可成长、可区分、可培养”的长期陪伴系统。

#### 核心范围
- 增加多技能
- 增加成长 / 养成
- 增加行为差异
- 让不同灵兽或不同培养方向具有实际区别

#### 关键拆解
1. 多技能
- 农业辅助
- 采集辅助
- 社交辅助
- 探索辅助

2. 成长 / 养成
- 亲密度
- 等级或阶段
- 技能解锁 / 技能强化
- 每日培养行为反馈

3. 行为差异
- 不同灵兽移动方式
- 不同互动偏好
- 不同协助倾向与节奏

#### 验收方向
- 至少 2 只灵兽具备不同能力与行为差异
- 成长结果会真实影响技能效果
- 玩家可以感知“养成前后”的变化

### 5. 更完整的 NPC 社交系统
#### 目标
将当前“基础对话 + 送礼反馈”升级为“日程差异、关系阶段、事件解锁”的可持续社交系统。

#### 核心范围
- 增加时间段对话差异
- 增加礼物偏好多维反馈
- 增加心事件 / 好感阶段解锁
- 让 NPC 关系推进形成明确目标感

#### 关键拆解
1. 时间段对话差异
- Morning / Afternoon / Evening / Night 文本差异
- 天气 / 节日 / 地点影响对话内容

2. 礼物偏好多维反馈
- loved / liked / neutral / disliked / hated
- 同类礼物重复赠送衰减
- 特定品质 / 特定类别礼物额外反馈

3. 心事件 / 好感阶段解锁
- 好感阶段门槛
- 特定条件触发事件
- 事件完成后解锁新对话 / 新奖励 / 新功能

#### 验收方向
- 至少 2 名 NPC 具备完整阶段式社交体验
- 时间段与礼物偏好会真实改变反馈
- 心事件可被稳定触发、记录、重置与存档
