# 战斗数据接入计划（JSON真值源）+ 元素双轨迁移 + 探索&战斗技能落地

> 目标：在不推翻现有实时轻战斗（能量+冷却）的前提下，把 **AuraElement**、**任务技能**、**灵兽技能** 做成“探索+战斗共用”的数据驱动系统，并最终把 battle 数据从 CSV 迁移到 JSON（ResourceManager 统一加载）。

---

## 0. 前置结论（来自现状）

- 当前战斗为 `engine::BattleManager/BattleField` 实时 Update 驱动。
- 项目已具备 `infrastructure::JsonValue::Parse`（轻量 JSON 解析）。
- `ResourceManager` 原本不支持文本；现已新增 `LoadText/GetText`（用于 JSON 读取与缓存）。
- 战斗已有 CSV 表：`spirit_table/skill_table/zone_table/weapon_table/balance_table`，短期不移除。

---

## 1. 里程碑拆分（建议按迭代落地）

### M1（1~2天）：JSON接入“骨架闭环”

- ✅ `ResourceManager`：新增 Text 通道（缓存+兜底）
- ✅ `BattleDataLoader`：读取并 Parse `battle/*.json`（先不映射字段）
- ⏳ `GameRuntime::Initialize`：调用 Loader 并记录日志（成功/失败）
- 交付标准：
  - JSON 缺失不崩溃（回退 CSV）
  - JSON 存在时可以被加载并 Parse（root 是 object）

### M2（2~4天）：AuraElement 双轨迁移（不破坏现有 ElementType）

- 引入 `domain::battle::AuraElement` 与 `ElementRules` 的运行时倍率表
- 在 `BattleField` 内部新增转换层：
  - `ElementType -> AuraElement`（legacy映射）
  - CSV 仍写 legacy element，JSON 写 aura element
- 逐步替换“倍率计算”使用 AuraElement（而不是旧五行/水火土）
- 交付标准：
  - 旧 CSV 不改也能跑
  - 新 JSON 若提供 aura element，倍率走新规则

### M3（3~6天）：任务技能（探索+战斗共用）最小实现

- 数据：`quest_skills.json`（6个技能）
- 存档：新增 `quest_skill|id|1` 行（只存解锁，不跨存档保留冷却）
- 探索触发点：
  - `GameRuntime::HandlePrimaryInteraction/HandleGiftInteraction/OnPlayerMoved` 中按需触发（例如灵眼高亮/风行步）
- 战斗触发点：
  - `BattleField::PlayerCastSkill` 前后插入“被动影响/主动施放入口”
- 交付标准：
  - 任务技能能被解锁（存档记忆）
  - 至少 1 个探索技能 + 1 个战斗技能可实测生效

### M4（4~8天）：灵兽技能（探索+战斗）接入 SpiritBeastSystem

- 数据：`pet_skills.json` + `pets.json`（或挂到既有灵兽卡）
- 战斗装配：进入战斗时从 SpiritBeastSystem 取心级/忠诚映射，生成 `BattlePartner`
- 探索光环：驻场/跟随时在 `GameRuntime` 的 tick 中提供效果（例如加速/预警）
- 交付标准：
  - 3只示例灵兽各2~3技能可运行
  - 战后疲劳/冷却逻辑不破坏现有战斗流程

## 3. AuraElement 双轨迁移策略（细则）

### 3.1 legacy映射（临时）

- `neutral -> neutral`
- `water/fire/wood/metal/earth -> cloud/wind/dew/glow/tide`（先按主题映射，后续可改）
- `light/dark -> light/dark`

### 3.2 CSV 与 JSON 的共存规则

- CSV：继续使用 legacy element（兼容历史表与代码）
- JSON：使用 aura element（新真值源）
- 运行时：
  - 若 JSON 提供 aura element：优先 aura
  - 否则：用 legacy→aura 映射

---

## 4. 探索&战斗技能“共用”定义

统一原则：技能数据只写一份（QuestSkillData/PetSkillData），运行时根据场景执行不同 effect handler。

- 探索 handler：移动速度、互动范围、高亮提示、触发概率加成
- 战斗 handler：净化倍率、控制、回能、护盾、标记

---

## 5. 风险与规避

- **风险：JsonValue 解析失败导致战斗不可用**
  - 规避：Loader 失败只 Warning，继续走 CSV
- **风险：ResourceManager 文本通道与 ref_counts_ 冲突**
  - 规避：文本 id 与纹理/字体/音效 id 命名分区（如 `battle_*_json`）
- **风险：探索技能触发点散落**
  - 规避：集中到 `GameRuntime` 的 SkillRuntime 子模块（后续可重构为 engine/systems）

## 6. 净化玩法升华的工程落地增补（v1.1）

### 6.1 机制增补（保持实时轻ATB）

- 战斗目标明确为“净化失衡”：
  - 保留 `pollution`，新增可选 `imbalance_segments`（5段）；
  - 段位清零即净化完成（MVP 可先“段位影响净化效率”，后续再升级为胜负条件）。
- 新增“同元素反噬”规则：
  - 命中同元素时触发 `segment_restore` 或 `extra_energy_cost`。
- 新增“茶道连携”：
  - 在 `BattleField` 维护最近 4 次技能 tag，命中序列触发 bonus。

### 6.2 代码接入建议

- `include/CloudSeamanor/engine/BattleEntities.hpp`
  - 可选新增 `EnemyImbalanceState`（segment values / highlighted segments）。
- `src/engine/BattleField.cpp`
  - 在 `ApplySkillHit_` 后追加段位处理；
  - 在 `UpdateSpiritAI_` 使用反噬和受击护栏；
  - 增加连携状态缓存与判定函数。
- `src/engine/GameRuntime.cpp`
  - 战败惩罚仅时间/生态影响，不扣核心资源；
  - 确保撤离入口常驻可用。

### 6.3 可玩性护栏（验收新增）

- 普通战斗平均时长 15~30 秒（500 次批量模拟）。
- 战斗操作点不超过 4。
- 可随时撤离且不出现“软锁死”。
- 战败后 1 分钟内可恢复到可玩状态（无永久损失）。
