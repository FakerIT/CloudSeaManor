# 《云海山庄》测试流程与标准总纲

**版本**: 1.0  
**制定日期**: 2026-04-18  
**适用范围**: 项目全生命周期测试  
**对标竞品**: 星露谷物语(Stardew Valley)

---

## 一、测试目的

### 1.1 核心目的
1. **验证功能完整性**: 确保每个系统按GDD设计文档正确实现
2. **保障用户体验**: 确保游戏体验达到"治愈系生活模拟"的定位目标
3. **对标竞品标准**: 确保核心系统达到或超越SDV的基础体验
4. **发现与修复缺陷**: 在发布前发现并解决所有P0/P1级别问题
5. **验证数值平衡**: 确保经济曲线、体力消耗、成长节奏符合设计意图

### 1.2 测试原则
- **无压力原则**: 测试过程本身也应遵循"无压力"理念,不追求一次全过
- **数据驱动原则**: 测试结论必须基于可量化的数据,而非主观感受
- **迭代原则**: 每次迭代都应重新测试受影响的功能点
- **对标原则**: 关键系统必须与SDV进行体验对比
- **回归原则**: 修复Bug后必须回归测试相关联的系统

## 二、测试阶段划分

### 2.1 阶段总览

| 阶段 | 名称 | 触发时机 | 测试重点 | 通过标准 |
|------|------|---------|---------|---------|
| T0 | 冒烟测试 | 每次构建后 | 核心循环可运行 | 游戏可启动并完成1个游戏日 |
| T1 | 功能测试 | 每个功能点完成时 | 单系统功能正确性 | 功能点100%通过验收标准 |
| T2 | 集成测试 | 每周/每Sprint | 系统间联动正确性 | 联动场景无阻断Bug |
| T3 | 体验测试 | 每个里程碑 | 整体游戏体验 | 目标用户可连续游玩30分钟+ |
| T4 | 对标测试 | 每月/重大版本 | 与SDV体验对比 | 核心系统体验不低于SDV |
| T5 | 发布测试 | 发布前 | 全面质量检查 | 无P0/P1 Bug |

### 2.2 各阶段详细说明

#### T0 冒烟测试(Smoke Test)
**目的**: 确认构建可用,核心循环可运行  
**执行者**: 开发者  
**频率**: 每次代码提交后  
**耗时**: 5-10分钟  

**检查项**:
- [ ] 游戏可正常启动(无崩溃)
- [ ] 主菜单可显示(如已实现)
- [ ] 可进入游戏场景
- [ ] 玩家可移动(WASD)
- [ ] 可与至少1个对象交互(E键)
- [ ] 游戏时钟正常推进
- [ ] 可正常退出

**通过标准**: 7/7项通过

---

#### T1 功能测试(Functional Test)
**目的**: 验证单个系统功能正确性  
**执行者**: 开发者/测试人员  
**频率**: 每个功能点完成时  
**耗时**: 30-60分钟/系统  

**执行流程**:
1. 从测试用例库中选取对应系统的测试用例
2. 按测试用例逐步执行
3. 记录实际结果与预期结果的差异
4. 标记通过/失败状态
5. 失败项创建Bug报告

**通过标准**: 
- P0用例: 100%通过
- P1用例: 95%通过
- P2用例: 80%通过

---

#### T2 集成测试(Integration Test)
**目的**: 验证系统间联动正确性  
**执行者**: 测试人员  
**频率**: 每周/每Sprint结束  
**耗时**: 2-4小时  

**关键联动场景**:
1. 种植→加工→售卖→送礼→好感(经济-社交闭环)
2. 云海→作物生长→灵茶品质→加工收益(天气-农业闭环)
3. 契约→解锁→建筑→灵气→云海(目标-经营闭环)
4. NPC日程→送礼→心事件→剧情(社交-叙事闭环)
5. 灵兽→浇水辅助→农活效率→好感(灵兽-农业闭环)
6. 季节→节日→预告→参与→奖励(时间-节日闭环)

**通过标准**: 6个闭环场景无阻断Bug

---

#### T3 体验测试(Experience Test)
**目的**: 验证整体游戏体验  
**执行者**: 目标用户(25-40岁,偏好SDV/动森)  
**频率**: 每个里程碑  
**耗时**: 30-60分钟/人  

**测试任务**:
1. 从零开始游玩,不提供任何指导
2. 完成第一个游戏日的所有活动
3. 尝试与NPC互动
4. 尝试种植作物
5. 尝试使用工坊
6. 完成第一个节日

**评估维度**:
| 维度 | 评估标准 | 评分 |
|------|---------|------|
| 易上手性 | 新玩家能否在5分钟内理解基本操作 | 1-5分 |
| 核心循环感 | 玩家是否清楚"今天该做什么" | 1-5分 |
| 治愈感 | 玩家是否感到放松而非焦虑 | 1-5分 |
| 成就感 | 玩家是否获得正向反馈 | 1-5分 |
| 继续意愿 | 玩家是否想继续游玩 | 1-5分 |

**通过标准**: 平均分≥3.5分,无维度低于2.5分

---

#### T4 对标测试(Benchmark Test)
**目的**: 与SDV进行体验对比  
**执行者**: 熟悉SDV的测试人员  
**频率**: 每月/重大版本  
**耗时**: 4-8小时  

**对比维度**:
| 维度 | SDV基准 | 云海山庄目标 | 测试方法 |
|------|---------|-------------|---------|
| 种植流畅度 | 8/10 | ≥7/10 | 同场景对比操作流畅度 |
| 社交深度 | 9/10 | ≥6/10 | 完成一次完整送礼+对话流程 |
| 探索丰富度 | 8/10 | ≥5/10 | 完成一次灵界探索 |
| 经济平衡 | 7/10 | ≥7/10 | 模拟1年经济曲线 |
| 治愈体验 | 8/10 | ≥8/10 | 30分钟连续游玩体验 |
| 长期目标感 | 8/10 | ≥7/10 | 契约系统完成度追踪 |

**通过标准**: 6个维度平均≥6.5分

---

#### T5 发布测试(Release Test)
**目的**: 发布前全面质量检查  
**执行者**: 全团队  
**频率**: 发布前  
**耗时**: 1-2天  

**检查清单**:
- [ ] 所有P0 Bug已修复并验证
- [ ] 所有P1 Bug已修复或已记录为已知问题
- [ ] 冒烟测试100%通过
- [ ] 功能测试P0/P1用例100%通过
- [ ] 集成测试6大闭环无阻断
- [ ] 体验测试平均分≥3.5
- [ ] 对标测试平均分≥6.5
- [ ] 性能测试: 稳定60FPS(1280x720)
- [ ] 存档兼容性: 旧存档可正常加载
- [ ] 无硬编码路径/调试代码残留
- [ ] 所有文本无乱码/错字
- [ ] 音频无爆音/缺失

**通过标准**: 全部通过

## 三、测试标准体系

### 3.1 Bug严重等级定义

| 等级 | 定义 | 示例 | 修复时限 |
|------|------|------|---------|
| P0-致命 | 游戏崩溃/数据丢失/核心功能完全不可用 | 启动崩溃、存档损坏、无法移动 | 立即(24h内) |
| P1-严重 | 核心功能异常但可绕过/严重影响体验 | NPC不出现、作物不生长、无法保存 | 3天内 |
| P2-一般 | 非核心功能异常/体验降级但不阻断 | UI显示错误、音效缺失、数值偏差 | 1周内 |
| P3-轻微 | 视觉瑕疵/文案错误/体验优化 | 像素偏移、错别字、动画不流畅 | 下个版本 |
| P4-建议 | 非Bug,改进建议 | 增加快捷键、优化动画 | 评估后决定 |

### 3.2 功能验收标准模板

每个功能点必须定义以下验收标准:

```
功能名称: [功能名]
所属系统: [系统名]
GDD参考: [GDD章节]
优先级: [P0/P1/P2]

验收标准:
1. [标准1 - 必须满足]
2. [标准2 - 必须满足]
3. [标准3 - 应该满足]

边界条件:
- [边界1]
- [边界2]

异常处理:
- [异常1] → [预期行为]
- [异常2] → [预期行为]

性能要求:
- [响应时间]
- [内存占用]

关联系统:
- [关联系统1]
- [关联系统2]
```

| 系统 | 验证指标 | GDD目标 | 允许偏差 |

## 四、测试流程

### 4.1 日常测试流程

```
代码提交 → T0冒烟测试 → 通过? 
    → 是: 标记构建可用
    → 否: 回退代码,修复后重新提交
```

### 4.2 功能测试流程

```
功能开发完成 → 自测(开发者) → 提交测试(测试人员) 
    → T1功能测试 → 通过?
        → 是: 标记功能完成,进入T2集成测试队列
        → 否: 记录Bug → 修复 → 重新测试(回归)
```

### 4.3 Sprint测试流程

```
Sprint结束前2天 → T2集成测试 → 通过?
    → 是: Sprint可关闭
    → 否: 记录Bug → 评估影响 → 修复/延期
```

### 4.4 里程碑测试流程

```
里程碑到达 → T3体验测试 + T4对标测试 → 通过?
    → 是: 进入下一阶段
    → 否: 生成改进方案 → 修复 → 重新测试
```

### 4.5 发布测试流程

```
发布前1周 → T5发布测试 → 通过?
    → 是: 准备发布
    → 否: 修复 → 重新T5 → 通过后发布
```

## 七、测试指标与KPI

### 7.1 质量指标

| 指标 | 目标值 | 计算方式 |
|------|--------|---------|
| 冒烟测试通过率 | 100% | 通过次数/总构建次数 |
| 功能测试通过率 | ≥95% | 通过用例数/总用例数 |
| P0 Bug修复率 | 100% | 已修复P0数/总P0数 |
| P1 Bug修复率 | ≥95% | 已修复P1数/总P1数 |
| Bug reopen率 | ≤10% | 重新打开Bug数/已关闭Bug数 |
| 存档兼容率 | 100% | 可加载旧存档数/总旧存档数 |

### 7.2 体验指标

| 指标 | 目标值 | 测量方式 |
|------|--------|---------|
| 新手上手时间 | ≤5分钟 | 首次游玩完成第一个交互的时间 |
| 核心循环完成率 | ≥80% | 玩家完成种植→收获→售卖的比例 |
| 30分钟留存率 | ≥70% | 游玩30分钟后仍想继续的比例 |
| 崩溃率 | ≤0.1% | 崩溃次数/总游玩小时数 |
